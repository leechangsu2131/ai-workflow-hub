# RAG 고도화 계획 (유튜브 마스터클래스 기반)

> "초보자를 위한 RAG 개념 20분 마스터클래스"를 바탕으로 `school-agent`의 문서 인덱싱 스킬(`doc-indexer`)을 고도화하는 플랜입니다.

---

## 1. 현재 `school-agent` (Phase 1)의 한계점
현재 구현된 `indexer.py`와 `search.py`는 **가장 기본적인 1세대 RAG 구조**입니다.
- **Dense Embedding 단독 사용**: `ko-sroberta-multitask`로 의미 기반 유사도만 검색 (Semantic Search).
- **단순 청킹 (Length-based)**: 500자 단위로 무조건 자르기 때문에 문단 중간에서 의미가 끊김.
- **맥락 손실**: 청크만 보면 이게 어느 학교, 어느 부서 기안인지 알 수 없음.

---

## 2. 유튜브 내용 기반 고도화 전략 (Phase 1.5)

### A. 하이브리드 서치 (Hybrid Search) 도입
- **이유**: 교육/행정 공문에는 "**현장체험학습**", "**NEIS**", "**품의서**" 같은 고유 명사와 키워드가 매우 중요함. 의미 유사도(Dense)만으로는 정확한 키워드가 포함된 문서를 놓칠 수 있음.
- **구현 방향**: 
  - 키워드 매칭(Sparse) 역할을 할 `BM25` (또는 `Elasticsearch`) 추가 도입.
  - 검색 시 Dense(ChromaDB) 결과와 Sparse(BM25) 결과를 `EnsembleRetriever`로 합산(Reciprocal Rank Fusion, RRF)하여 최종 순위 산출.

### B. 스마트 청킹 (표/구조 보존)
- **이유**: "여러분의 청크가 정크면 어떤 모델을 써도 소용없다." 공문에서 가장 중요한 데이터는 '표(Table)'에 있음. 단순 500자 자르기로는 표 데이터가 박살남.
- **구현 방향**: 
  - `odfpy` 파싱 시 단락(Paragraph)과 표(Table)를 구분하여 구조 기반 청킹 적용. (LangChain의 `MarkdownTextSplitter`나 문서 요소 기반 청킹 활용)

### C. 컨텍스추얼 리트리벌 (Contextual Retrieval)
- **이유**: 청크만 떼어놓고 보면 의미를 알 수 없는 현상 방지. "전년 대비 10% 증액"이라는 청크가 '어느 연도, 어느 예산'인지 잃어버리지 않게 함.
- **구현 방향**: 
  - 인덱싱 단계에서 성능이 좋은 LLM(예: Claude Haiku 등 저렴한 모델)을 사용하여, **각 청크 앞에 문서의 전체 요약을 1~2줄 요약으로 붙여서 벡터화**함.
  - *비용이 문제라면 유튜브 팁처럼 "정보가 있는 청크에만" 적용.*

### D. 메타데이터 필터링 강화
- **구현 방향**: 단순히 날짜만 넣는 게 아니라 파일명에서 '연도', '부서(교무/연구/과학/체육 등)'를 정규식으로 추출하여 ChromaDB 메타데이터로 저장. 
- 검색 시 "2023년", "과학부" 등의 조건 필터링을 먼저 거친 후 검색.

---

## 3. 적용 프로세스 제안

바이브 코딩으로 다음 순서대로 `doc-indexer`를 업그레이드하는 것을 추천합니다.

1. **[Step 1] Metadata & Chunking 개선**: `indexer.py`의 청킹 로직을 개선하고 연도/부서 메타데이터 자동 추출 추가.
2. **[Step 2] BM25 도입 (Hybrid Search)**: `rank_bm25` 패키지 설치. `search.py`에서 ChromaDB 유사도 + BM25 키워드 매칭을 융합하는 로직 작성.
3. **[Step 3] Re-ranker 도입**: 하이브리드 검색으로 뽑은 Top 10을 `BGE-Reranker` 같은 크기가 작은 로컬 리랭크 모델(Cross-Encoder)로 다시 순위 매기기.

> **💡 조언**: 컨텍스추얼 리트리벌은 LLM API 비용과 속도 저하가 발생하므로, 학교 업무 자동화를 개인 로컬에서 돌린다면 일단 **[하이브리드 서치 + 메타데이터 필터 + 리랭커]** 세트까지만 구현해도 성능이 기하급수적으로 올라갑니다.
